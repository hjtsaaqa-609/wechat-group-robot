# DAS 接口异常排查说明

更新时间：2026-04-14

## 背景

本项目在生成《DAS 周报》时，依赖 DAS 平台以下接口：

- `GET /api/inspection-stats`
- `GET /api/cleaning-quality`

在 2026-04-14 本地多次执行周报 `dry-run` 过程中，发现上述接口存在异常，导致部分 KPI 无法稳定展示。

当前受影响指标：

- 年度累计异常组件解决率
- 清洁度未达预期率

## 一、`/api/inspection-stats` 异常情况

### 1. 现象

`/api/inspection-stats` 并不是所有区间都会失败。

当前已验证结果：

- `2026-04-06` 到 `2026-04-12`：可正常返回
- `2026-04-01` 到 `2026-04-14`：可正常返回
- `2026-03-01` 到 `2026-03-31`：可正常返回
- `2026-02-01` 到 `2026-02-28`：可正常返回
- 任何包含 `2026-01` 的查询区间：返回 `502 Bad Gateway`

已实际失败的区间包括：

- `2026-01-01` 到 `2026-01-31`
- `2026-01-01` 到 `2026-01-07`
- `2026-01-08` 到 `2026-01-14`
- `2026-01-08` 单天
- `2026-01-09` 单天
- `2026-01-01` 到 `2026-04-14` 年累计区间

### 2. 结论

当前判断这不是“查询跨度太大”导致的问题，而是：

- `inspection-stats` 只要查询区间覆盖 `2026-01`，就会触发后端 `502`

因此，年度累计异常组件解决率无法正常取数，不是前端报表逻辑问题，而是 DAS 后端在处理 `2026-01` 巡检统计数据时发生异常。

### 3. 周报影响

周报中的以下指标当前被迫降级：

- `年度累计异常组件解决率`

当前展示策略：

- 接口成功：展示实际值
- 接口失败：显示 `尚未入系统（DAS inspection-stats 只要查询区间包含 2026-01 即会返回 502）`

### 4. 建议 DAS 后端排查方向

- 排查 `2026-01` 巡检数据是否存在脏数据、空字段或异常状态值
- 排查 `inspection-stats` 在 `2026-01` 区间内的聚合 SQL / ORM 查询是否报错
- 排查是否存在单条记录导致整段统计失败
- 建议直接先用最小区间复现：
  - `2026-01-08`
  - `2026-01-09`
  - `2026-01-01 ~ 2026-01-07`

### 5. 典型报错

```text
DAS 请求失败: GET /api/inspection-stats?start_at=2025-12-31T16%3A00%3A00.000Z&end_at=2026-04-14T15%3A59%3A59.999Z | 502 Bad Gateway
```

## 二、`/api/cleaning-quality` 异常情况

### 1. 现象

周报中的“清洁度未达预期率”已改为使用清扫质量页面口径，也就是：

- `GET /api/cleaning-quality`
- 指标取值：`summary.belowExpectedRatio`
- 统计维度：电站维度

同一统计周期 `2026-04-06 ~ 2026-04-12` 曾出现以下两种情况：

- 某次单独查询成功，返回：
  - `participatingSiteCount = 108`
  - `belowExpectedSiteCount = 55`
  - `belowExpectedRatio = 0.5092592592592593`
- 某次在完整周报执行链路中返回：
  - `502 Bad Gateway`

### 2. 结论

`/api/cleaning-quality` 当前更像是“间歇性失败”问题，而不是固定区间、固定日期必现的脏数据问题。

与 `inspection-stats` 不同，`cleaning-quality` 已经证明同一时间范围存在成功返回，因此目前判断优先级如下：

- 优先怀疑接口或网关稳定性问题
- 其次排查查询负载、并发、超时、缓存或上游依赖波动

### 3. 周报影响

周报中的以下指标会受影响：

- `清洁度未达预期率`

当前展示策略：

- 接口成功：展示实际值
- 接口失败：显示 `--`

### 4. 建议 DAS 后端排查方向

- 排查 `/api/cleaning-quality` 是否存在偶发超时
- 排查网关层或反向代理层是否提前返回 `502`
- 排查接口在并发调用下是否更容易失败
- 建议重点关注区间：
  - `2026-04-06 ~ 2026-04-12`

### 5. 成功样例

```json
{
  "participatingSiteCount": 108,
  "belowExpectedSiteCount": 55,
  "belowExpectedRatio": 0.5092592592592593
}
```

### 6. 典型报错

```text
DAS 请求失败: GET /api/cleaning-quality?start_at=2026-04-05T16%3A00%3A00.000Z&end_at=2026-04-12T15%3A59%3A59.999Z | 502 Bad Gateway
```

## 三、对周报系统的当前处理方式

为避免错误数据直接进入周报，当前报表逻辑采用降级展示：

- `年度累计异常组件解决率`
  - 接口失败时显示明确原因说明
- `清洁度未达预期率`
  - 接口失败时显示 `--`

这样做的目的是：

- 不误报错误 KPI
- 保留问题定位线索
- 便于后续在 DAS 修复后恢复自动展示

## 四、建议后续动作

1. DAS 后端先修复 `inspection-stats` 对 `2026-01` 区间的查询异常。
2. DAS 后端排查 `cleaning-quality` 的间歇性 `502`。
3. 修复后，用以下两个区间回归验证：
   - `inspection-stats`：`2026-01-01 ~ 2026-04-14`
   - `cleaning-quality`：`2026-04-06 ~ 2026-04-12`
4. 验证通过后，再重新生成最终版 KPI 周报。
